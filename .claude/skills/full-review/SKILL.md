# NEXT-UP 백엔드 전수 검토 (파괴적 토론)

7개 전문 에이전트가 코드베이스를 전수 검토하고, SendMessage로 직접 상호 비판하며 토론한 뒤, 통합 리포트를 생성하는 스킬이다.

핵심 원칙:
1. **에이전트들이 1인 다역이 아니라 실제로 SendMessage를 통해 직접 의견을 주고받으며 토론해야 한다.**
2. **"카톡으로 하면 된다"는 논거 금지** — 이 서비스의 목표는 카톡 없이 팀 운영을 완결시키는 것이다. 기존에 카톡으로 하던 것을 앱 안에서 해결해야 한다.

---

## 실행 순서

### Phase 1: 코드베이스 구조 파악

스킬 실행 시 먼저 현재 코드베이스 규모를 파악한다:
- 소스 파일 수 (main/test)
- DB 마이그레이션 파일 수
- 모듈 구조 확인

### Phase 2: 팀 구성 및 에이전트 스폰

TeamCreate로 `backend-review` 팀을 생성한 뒤, 7개 에이전트를 **동시에** Agent 도구로 스폰한다.

모든 에이전트에 `team_name: "backend-review"`와 `model: "opus"`를 지정한다.

#### 에이전트 역할

| 에이전트 | name | 미션 |
|---------|------|------|
| **Fat-Trimmer** | `fat-trimmer` | 사회인 야구 현실에 맞지 않는 프로야구 중심 오버엔지니어링을 찾아 전량 삭제 리스트 작성. 단, "카톡으로 하면 된다"는 삭제 근거로 사용 금지 — 이 서비스가 카톡을 대체해야 한다 |
| **Gap-Finder** | `gap-finder` | 사회인 야구 현장에서 필수인데 설계에서 누락된 기능 발굴. 서비스 완결성 관점(카톡 없이 팀 운영 가능해야 함)에서 검토 |
| **DB-Query** | `db-query` | DB 마이그레이션 충돌, 인덱스 설계, N+1 문제, CASCADE 위험, 데이터 정합성 검증 |
| **Security-Auth** | `security-auth` | IDOR 취약점, @PreAuthorize 누락, Entity 직접 노출, 동시성 충돌, OWASP Top 10 전수 조사 |
| **Architecture-SRP** | `architecture-srp` | God Entity/Service, SRP 위반, Hexagonal 위반, Reflection 남용, DTO 패턴 불일치 비판 |
| **UX-Flow** | `ux-flow` | 사용자 동선 검증. "이 기능을 앱에서 몇 탭으로 완료할 수 있는지?", API 설계가 프론트엔드 화면 흐름에 맞는지, 카톡 대체를 위해 부족한 UX 흐름 식별 |
| **Baseball-Advisor** | `baseball-advisor` | 사회인 야구 현장 자문역. 실제 사회인 야구 리그 운영 경험 기반으로 규칙 정합성 검증, 현장에서 진짜 쓰이는 기능 vs 안 쓰이는 기능 판별, 다른 에이전트의 주장이 현장 현실과 맞는지 자문 |

#### 사전 결정사항 로딩

에이전트를 스폰하기 전에 `references/prior-decisions.md`를 읽어라. 이 파일에는 이전 검토에서 사용자와 합의된 확정 사항(삭제 대상, 유지 대상, 개념 변경)이 있다. 에이전트 프롬프트에 이 내용을 포함하여, 이미 합의된 것을 다시 논쟁하지 않고 **새로운 발견에 집중**하도록 지시하라.

#### 에이전트 프롬프트 필수 포함 사항

각 에이전트 프롬프트에 반드시 포함할 것:

1. **프로젝트 구조 안내**: 6개 모듈(core, infrastructure, api, backoffice, scorer, common)과 각 경로
2. **서비스 목표**: "카톡 없이 팀 운영을 완결시키는 사회인 야구 기록 서비스" — 카톡으로 대체 가능하다는 논거 금지
3. **사회인 야구 현실 컨텍스트**: 주말 리그, 9명 겨우 모으는 팀, 기록원 1명, 비전문 심판, 5~7이닝 경기, 인원 부족 빈번
4. **코드를 실제로 읽으라는 지시**: Glob, Grep, Read 도구로 실제 파일을 읽고 구체적인 파일명, 클래스명, 메서드명을 근거로 제시
5. **토론 규칙**: "예의는 버려라. 기술적 생존만 생각하라. 타협하지 마라."
6. **SendMessage 지시**: 분석 완료 후 다른 에이전트들에게 SendMessage로 결과를 공유하고, 다른 에이전트의 메시지를 받으면 기술적으로 반박하거나 동의하라
7. **다른 에이전트 목록**: 팀 내 다른 6개 에이전트의 name과 역할을 알려줘서 누구에게 어떤 질문/반박을 해야 하는지 알게 하라

### Phase 3: 분석 대기 (타임아웃 관리)

7개 에이전트가 동시에 코드베이스를 분석한다. 각 에이전트가 분석을 완료하고 팀에 결과를 공유할 때까지 대기한다.

**에이전트 장애 대응:**
- 10분 이상 응답 없는 에이전트는 SendMessage로 상태 확인
- 확인 후에도 응답 없으면 해당 에이전트를 종료하고 재스폰
- 2회 실패 시 해당 역할 없이 진행 (나머지 에이전트의 분석으로 리포트 작성)
- 사용자에게 장애 에이전트와 사유를 알린다

### Phase 4: 상호 비판 토론 촉발

7개 에이전트 전원이 분석을 완료하면, 팀 리드(나)가 각 에이전트에게 SendMessage로 토론을 시작하도록 지시한다.

각 에이전트에게 다른 에이전트의 핵심 발견을 요약하고, 구체적인 반박/동의 포인트를 제시하여 토론을 촉발한다:

- **fat-trimmer에게**: gap-finder의 반박, security-auth의 "보안만 추가하면 되는 것" 논거, baseball-advisor의 현장 검증 결과 전달
- **gap-finder에게**: fat-trimmer의 삭제 리스트 중 반박할 것, ux-flow의 UX 관점 누락 기능, baseball-advisor의 현장 필수 기능 확인
- **db-query에게**: fat-trimmer 삭제 시 DB 영향, architecture-srp의 N+1 지적 교차 검증
- **security-auth에게**: gap-finder 추가 기능의 보안 리스크, CASCADE + IDOR 결합 시나리오
- **architecture-srp에게**: fat-trimmer 삭제의 아키텍처 영향, gap-finder 추가의 수용 가능성, ux-flow가 요구하는 API 변경의 구조적 영향
- **ux-flow에게**: fat-trimmer가 삭제하려는 기능 중 UX상 필요한 것, gap-finder가 추가하려는 기능의 화면 흐름 제안, baseball-advisor의 현장 워크플로우와 앱 UX의 갭
- **baseball-advisor에게**: fat-trimmer의 삭제 리스트가 현장에서 맞는지, gap-finder의 추가 리스트 중 현장 우선순위, ux-flow의 UX 제안이 현장 사용성에 맞는지

에이전트들이 **직접 서로에게 SendMessage를 보내며** 토론하도록 한다.

**토론 수렴 기준:**
- 최소 1라운드 반박/재반박이 완료되어야 함
- 각 에이전트가 최소 1번 이상 다른 에이전트에게 직접 SendMessage를 보내야 함
- 새로운 발견 없이 기존 주장 반복이 시작되면 수렴으로 판단
- 최대 3라운드까지 허용 (그 이상은 수확 체감)

### Phase 5: 통합 리포트 작성

토론이 수렴하면 모든 에이전트를 종료(shutdown_request)하고, 다음 구조의 통합 리포트를 작성한다:

**파일 경로**: `outputs/reports/backend-review-{날짜}.md`

```markdown
# NEXT-UP 백엔드 전수 검토 통합 리포트

## 최종 판정: PASS / FAIL

## [1] 전량 삭제 리스트
### 만장일치 삭제 (전원 동의)
### 조건부 삭제 (토론 후 합의)
### 삭제 거부 (방어 성공)
### 삭제 철회 (양보)

## [2] 필수 추가 리스트
### CRITICAL — 현장 운영 불가 수준
### IMPORTANT — 서비스 품질
### DEFERRED — 선행 조건 미충족

## [3] 구조 개편 제안
### Phase 0 — 배포 차단 해제 (즉시)
### Phase 1 — 보안 긴급 패치
### Phase 2 — DB 방어 + 핵심 누락 기능
### Phase 3 — 아키텍처 리팩토링
### Phase 4 — 기능 확장

## [4] UX 개선 제안
### 카톡 대체를 위한 필수 UX 흐름
### API 변경이 필요한 화면 흐름

## [5] 작업 로드맵 (에이전트 합의)

## [6] 에이전트 간 주요 쟁점 요약
| 쟁점 | 공격 측 | 방어 측 | 결론 |

## [7] 양호한 점 (공통 인정)

## [8] GitHub 이슈 분류
모든 발견 사항을 GitHub 이슈로 변환할 수 있도록 분류한다.
각 이슈에 제목, 타입, 우선순위, 예상 작업 규모, 선행 조건을 포함한다.

### 타입별 분류
- 🔴 bug: 보안 취약점, DB 정합성 깨짐, 배포 불가 등 현재 동작이 잘못된 것
- 🟠 refactor: God Entity 분리, 패턴 통일, 죽은 코드 삭제 등 구조 개선
- 🟢 feature: 누락 기능 추가, 리디자인 필요 기능
- 🔵 chore: 마이그레이션 정리, 설정 변경 등

### 우선순위별 분류
- P0 (즉시): 배포 차단, 보안 CRITICAL
- P1 (1주): 보안 HIGH, 현장 운영 불가
- P2 (2주): DB 방어, 핵심 누락 기능
- P3 (3-4주): 아키텍처 리팩토링
- P4 (이후): 기능 확장

### 이슈 목록 테이블
| # | 제목 | 타입 | 우선순위 | 규모 (S/M/L/XL) | 선행 이슈 | Phase |
```

### Phase 6: 사용자 피드백 반영 + 사전 결정사항 갱신

리포트를 사용자에게 제시한 뒤, 사용자가 수정/추가/삭제를 요청하면:

1. **리포트 업데이트** — 사용자의 피드백을 리포트에 반영한다
2. **prior-decisions.md 자동 갱신** — 사용자가 확정한 결정(삭제/유지/개념 변경)을 `references/prior-decisions.md`에 추가한다. 다음 `/full-review` 실행 시 에이전트들이 이 결정을 전제로 분석하게 된다.

갱신 기준:
- 사용자가 "이건 삭제해" → 확정된 삭제 대상 테이블에 추가
- 사용자가 "이건 유지해" / "이건 필요해" → 확정된 유지 대상 테이블에 추가
- 사용자가 개념을 교정함 ("용병은 그게 아니라...") → 확정된 개념 변경 테이블에 추가
- 사용자가 새로운 제품 원칙을 제시함 → 확정된 제품 원칙에 추가

## 사회인 야구 도메인 컨텍스트

에이전트들이 분석할 때 참고해야 할 사회인 야구 현실:

- **서비스 목표**: 카톡 없이 팀 운영을 완결시키는 사회인 야구 기록 서비스. 투표, 공지, 일정, 정산, 기록, 소통 모두 이 앱 안에서 해결
- **팀 구성**: 주말에 9명 겨우 모으는 팀이 대부분. 8명만 올 때도 빈번
- **기록원**: 1명이 전부. 매 투구 실시간 기록은 선택 기능 (기록 능력 있는 팀은 타자별 구종 분석 활용 가능)
- **심판**: 비전문 심판. 인필드플라이 등 복잡한 판정은 드묾
- **경기**: 5~7이닝이 보통. 시간 제한 있는 경기가 많음
- **용병/이벤트 게임**: 팀 상관 없이 야구하고 싶은 사람들이 모여서 하는 픽업 게임/올스타전 개념. "팀에 임시 선수 빌려오기"가 아님
- **이적**: 개념 자체가 없음. 탈퇴하고 새 팀 가입하면 끝
- **팀 운영**: 투표(경기/연습/회식/모임) 통합, 경기 1주일 전 자동 투표 생성 + 하루 전 자동 마감
- **징계**: 영구 제재(BAN)만 시스템 필요
- **구장**: 예약과 양도 모두 앱 안에서 처리 (카톡 양도 X)

## 주의사항

- 에이전트를 스폰할 때 반드시 `team_name`과 `name` 파라미터를 지정하여 SendMessage로 통신 가능하게 할 것
- 1인 다역 금지 — 반드시 실제 에이전트 7개를 스폰하여 독립적으로 코드를 읽고 분석하게 할 것
- 에이전트 간 토론은 팀 리드가 중개하지 않고, 에이전트들이 **직접** 서로에게 SendMessage를 보내야 함
- 토론이 충분히 진행된 후에만 리포트를 작성할 것 (최소 1라운드 반박/재반박)
- **"카톡으로 하면 된다"는 삭제/축소 근거로 절대 사용 금지** — 이 서비스가 카톡을 대체해야 함
- baseball-advisor의 현장 자문은 다른 에이전트의 주장에 대한 최종 현실 검증 역할 — 현장에서 진짜 필요한지 판별하는 결정적 근거
- 리포트 작성 후 에이전트 종료를 잊지 말 것 (shutdown_request)
