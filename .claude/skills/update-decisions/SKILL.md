# 사전 결정사항 갱신

사용자의 제품 결정을 `full-review` 스킬의 레퍼런스 파일(`references/prior-decisions.md`)에 반영하는 스킬이다.

---

## 실행 흐름

### 1. 사용자 의견 파악

사용자가 이 스킬을 실행하면 함께 전달한 메시지에서 결정 사항을 파악한다.

결정 유형:
- **삭제**: "XX 삭제해", "XX 필요없어", "XX 지워"
- **유지**: "XX 유지해", "XX 필요해", "XX 살려"
- **개념 변경**: "XX는 그게 아니라 YY야", "XX를 YY로 바꿔야해"
- **제품 원칙**: "이 서비스는 XX해야해", "XX는 앱에서 다 해결해야해"
- **리디자인**: "XX를 YY 방식으로 바꿔야해"
- **조회**: "지금까지 결정한 거 보여줘", "현재 결정 목록", "prior-decisions 보여줘"

조회 요청 시에는 prior-decisions.md의 현재 내용을 읽어서 섹션별로 요약해서 보여준다.

### 2. prior-decisions.md 읽기

파일 경로: `.claude/skills/full-review/references/prior-decisions.md`

현재 내용을 읽어서 중복 확인한다. 이미 같은 결정이 있으면 사용자에게 알리고 업데이트 여부를 확인한다.

### 3. 적절한 섹션에 추가

| 결정 유형 | 추가 위치 |
|----------|----------|
| 삭제 결정 | `## 확정된 삭제 대상` 테이블에 행 추가 |
| 유지 결정 | `## 확정된 유지/리디자인 대상` 테이블에 행 추가 |
| 개념 변경 | `## 확정된 개념 변경` 테이블에 행 추가 |
| 제품 원칙 | `## 확정된 제품 원칙` 리스트에 항목 추가 |
| 기존 결정 번복 | 해당 행을 삭제하고 새 결정을 올바른 섹션에 추가 |

### 4. 기존 결정 번복 감지

사용자의 새 결정이 기존 결정과 모순될 때 (예: 이전에 "유지"라고 한 것을 "삭제"로 변경):

1. 기존 결정을 보여주고 번복 의도를 확인한다
2. 확인되면 기존 결정을 제거하고 새 결정을 올바른 섹션에 추가한다

```
⚠️ 기존 결정과 충돌 감지:
- 기존: [유지] SeasonAward — MVP 투표는 팀원들에게 중요한 기능
- 신규: [삭제] SeasonAward

기존 결정을 번복하고 삭제 대상으로 변경할까요?
```

### 5. 변경 요약 출력

변경 내용을 사용자에게 **항상 동일한 형식으로** 보여준다:

```
prior-decisions.md 업데이트 완료:

📋 변경 내역:
- [추가] 확정된 삭제 대상: BookingTransfer (구장 예약 양도 거래)
  사유: 상태머신+가격 과잉. 단순 양도 기능으로 재설계

📁 파일: .claude/skills/full-review/references/prior-decisions.md
```

응답은 반드시 다음을 포함한다:
- 변경 유형 ([추가], [수정], [이동], [삭제])
- 대상 섹션명
- 대상 항목명
- 사유 (사용자가 말한 이유)
- 파일 경로

### 5. full-review 스킬의 도메인 컨텍스트도 동기화

결정 사항이 `full-review/SKILL.md`의 "사회인 야구 도메인 컨텍스트" 섹션과 충돌하면 해당 섹션도 업데이트한다.

예: 사용자가 "구장 양도는 앱에서 해야해"라고 결정 → 도메인 컨텍스트의 "구장" 항목도 갱신

## 사용 예시

```
/update-decisions PitchEvent는 유지. 기록 능력 있는 팀은 쓸 수 있게.
/update-decisions 이적 개념 없어. 탈퇴하고 새 팀 가입이면 끝.
/update-decisions 투표는 경기/연습/회식/모임 카테고리로 통합해야해.
/update-decisions Election 삭제 확정. transferOwnership으로 대체.
```

---

## 주의사항

- prior-decisions.md가 없으면 `full-review/references/` 디렉토리에 새로 생성한다
- 기존 결정과 모순되는 새 결정이 들어오면, 기존 결정을 제거하고 새 결정으로 교체한다 (사용자에게 알림)
- 테이블 형식(마크다운)을 유지한다
- 결정의 "사유" 컬럼에는 사용자가 말한 이유를 간결하게 기록한다
